# Agent 工具链与评测基础设施：技术分层、商业化路径与投资观察

**公开资料研究样稿｜彭德东｜2026 年 7 月**

> 本文用于整理记录人工智能经济学组会的Agent部分调研结果与观点，不构成投资建议。

## 一、核心结论

### 判断 1：Agent 框架正在从“能力展示层”下沉为基础组件，价值重心转向运行时、评测与企业集成

Agent 产品早期常用单次 Demo 证明“模型会调用工具”。进入企业工作流后，问题变成了长任务能否恢复、状态是否可追溯、失败能否回滚、敏感动作能否审批，以及版本更新后性能是否退化。LangGraph 将 durable execution、persistence 与 human-in-the-loop 列为核心能力，说明编排框架本身也在向运行时靠拢。[LangGraph 官方文档](https://docs.langchain.com/oss/python/langgraph/overview)

**投资含义：** 单纯封装模型与提示词的产品迁移成本较低。更值得验证的壁垒包括：与客户系统的深度连接、真实任务轨迹形成的评测数据、权限与审计体系，以及围绕异常处理沉淀的工作流知识。

### 判断 2：MCP 与 A2A 降低连接成本，但协议兼容不等于产品壁垒

MCP 通过标准接口暴露 tools、resources 与 prompts；A2A 则面向独立 Agent 之间的消息、任务与产物交换。二者解决的分别是“Agent 如何连接外部能力”和“Agent 如何与其他 Agent 协作”。[MCP 规范](https://modelcontextprotocol.io/specification/2025-06-18/server/tools)｜[A2A 规范](https://a2a-protocol.org/v1.0.0/)

**投资含义：** 协议普及会降低基础连接器的稀缺性。公司的防御力不能只用“支持多少 MCP Server”衡量，还要看身份与权限映射、数据语义适配、调用质量、跨系统事务一致性，以及对高风险动作的治理能力。

### 判断 3：Agent 评测必须从终局成功率扩展到轨迹质量与运营指标

AgentBench 将 Agent 放入多轮交互环境；AgentBoard 进一步提出 progress rate，尝试刻画任务过程中的增量进展；SWE-bench 及其后续版本则将代码 Agent 放到真实仓库问题中评估。这些工作共同表明，只看最终答案难以解释失败来自规划、工具、状态、环境还是恢复机制。[AgentBench](https://arxiv.org/abs/2308.03688)｜[AgentBoard](https://arxiv.org/abs/2401.13178)｜[SWE-bench Verified 说明](https://openai.com/index/introducing-swe-bench-verified/)

**投资含义：** 评测平台若只提供 LLM-as-a-judge 打分，差异化可能有限。更有价值的产品应能连接线上轨迹、回放失败、管理数据集版本，并把质量、成本、延迟和风险放在同一套发布门禁中。

### 判断 4：企业 Agent 的商业价值来自“可授权的行动闭环”，同时也放大治理需求

生成内容带来的价值容易停留在助手层。Agent 能读取业务状态并执行动作，才可能缩短流程周期、替代重复劳动；但执行权限越高，错误成本也越高。LangGraph 的 human-in-the-loop 设计允许对工具调用执行 approve、edit 或 reject，并依赖持久化状态实现暂停和恢复。[Human-in-the-loop 官方文档](https://docs.langchain.com/oss/python/langchain/human-in-the-loop)

**投资含义：** 研究产品时应同时问“能自动完成多少任务”和“错误动作如何被限制”。审批率下降并不必然代表产品成熟，也可能来自风险策略过松。

## 二、技术栈地图

| 层级 | 主要任务 | 关键工程问题 | 可验证指标 |
|---|---|---|---|
| 模型与推理 | 理解、规划、生成与工具选择 | 模型路由、缓存、上下文预算、结构化输出 | 单任务推理成本、首 token/总延迟、格式错误率 |
| Context 与数据 | 检索企业知识和实时状态 | 权限继承、召回、时效性、数据血缘 | 有权限召回率、答案可追溯率、过期信息率 |
| 工具与协议 | 连接 API、数据库和业务系统 | schema、幂等性、鉴权、限流、错误语义 | 工具成功率、重试率、不可逆错误率 |
| 编排与运行时 | 维护状态并协调多步骤任务 | checkpoint、暂停恢复、并发、超时与回滚 | 任务恢复率、平均重试次数、状态丢失率 |
| 记忆 | 保存跨轮或跨任务信息 | 写入策略、遗忘、隐私隔离、污染控制 | 有效命中率、错误记忆率、删除完成率 |
| 评测与可观测性 | 记录轨迹、回归测试、线上监控 | 数据集版本、失败归因、漂移、人工校验 | 任务成功率、轨迹完成度、回归告警命中率 |
| 安全与治理 | 控制权限和高风险动作 | 最小权限、审批、审计、提示注入防护 | 越权拦截率、人工干预率、审计覆盖率 |
| 应用与交付 | 嵌入具体业务工作流 | 流程重构、系统集成、用户采纳 | 部署周期、周活跃用户、节省工时、续费扩张 |

这一分层的重点不是把公司机械归类，而是找出价值捕获位置。例如，一家声称做“企业 Agent 平台”的公司，收入可能主要来自私有化集成；另一家提供评测产品，真正壁垒可能是客户持续回流的生产轨迹。两者的毛利结构、销售周期和可扩展性完全不同。

## 三、从 Demo 到生产：四个技术门槛

### 1. 状态与恢复

长任务包含模型调用、工具调用和外部副作用。若失败后从头重跑，成本会上升，还可能重复发送邮件或重复写入数据库。生产系统需要 checkpoint、幂等工具和清晰的恢复边界。LangGraph 的持久化机制会在执行步骤保存状态，并支持恢复、人工介入和故障容错。[Persistence 官方文档](https://docs.langchain.com/oss/python/langgraph/persistence)

尽调时应实际观察：断网、模型超时或工具返回脏数据后，系统从哪个节点恢复；已经成功的外部操作是否会被重复执行。

### 2. 权限与人工审批

“是否支持 human-in-the-loop”不是二元问题。需要继续拆解：

- 哪些工具默认需要审批；
- 审批人是否由企业 IAM 与业务角色决定；
- 审批时能否看到参数、上下文和风险原因；
- 修改参数后是否重新触发策略判断；
- 拒绝后系统是否真正停止后续副作用。

本作品集中的 Research Agent 专门设置了回归测试：审批状态为 `rejected` 时，不允许进入代码修改和执行节点。这个测试来自真实开发中发现的控制流风险，而不是为了展示框架 API。

### 3. 评测数据与发布门禁

静态评测集容易被提示词和模型版本过拟合。更稳健的体系应包含：

- 离线黄金任务：覆盖正常路径、边界条件与高风险反例；
- 轨迹级指标：计划质量、工具选择、参数正确性和恢复行为；
- 线上抽样：由业务专家复核真实任务；
- 新鲜任务：持续吸收尚未进入开发循环的样本；
- 发布门禁：质量下降或成本超限时阻止版本上线。

NIST AI Resource Center 将 testing、evaluation、verification 与 validation 作为 AI 风险管理落地的重要资源，说明评测也承担治理职能，而不仅是研发排名。[NIST AIRC](https://airc.nist.gov/)

### 4. 成本与延迟

Agent 的成本不能只看单次模型价格。更实用的计算单位是：

**单个成功任务全成本 =（模型成本 + 工具成本 + 基础设施成本 + 人工复核成本）÷ 经业务确认的成功任务数。**

如果增加规划轮数让成功率提高，却显著增加人工复核和长尾延迟，商业价值未必改善。成熟系统通常需要按任务风险进行模型路由，对重复上下文做缓存，并给循环设置预算和停止条件。

## 四、评测框架：建议同时看五类指标

### A. 结果质量

- 业务确认的任务成功率；
- 关键字段准确率与完整率；
- 与人工或原流程相比的增量提升。

### B. 过程质量

- 计划完成度；
- 无效工具调用率；
- 重试、回退和循环次数；
- 失败可归因比例。

### C. 运营效率

- 每个成功任务成本；
- p50/p95 完成时延；
- 人工干预率和平均复核时长；
- 单客户峰值并发与资源占用。

### D. 安全与治理

- 越权调用拦截率；
- 不可逆动作的审批覆盖率；
- 提示注入或脏数据攻击下的成功防护率；
- 审计日志完整率。

### E. 商业结果

- 从签约到首个生产流程的周期；
- 活跃工作流数量与调用深度；
- 客户续费、席位或用量扩张；
- 部署后实际节省工时或缩短流程周期。

其中最重要的组合指标是“成功任务成本”和“业务确认成功率”。只看 token 消耗可能奖励低质量系统，只看成功率则可能忽略无法规模化的人工兜底。

## 五、商业化路径与竞争壁垒假设

### 路径 1：开发框架与运行时

常见模式包括开源核心、云托管、按调用计费和企业版。开源有助于获得开发者分发，但商业化要验证托管渗透率、生产工作负载比例和企业功能付费意愿。

**潜在壁垒：** 稳定运行时、生态集成、迁移成本和生产轨迹。  
**反证信号：** 下载量高但付费工作负载少；客户只把产品用于原型。

### 路径 2：评测与可观测性

产品可按席位、追踪量、评测任务量或企业部署收费。其价值来自帮助团队缩短定位与发布周期。

**潜在壁垒：** 跨框架数据模型、失败案例库、评测数据网络效应和发布流程集成。  
**反证信号：** 主要功能被云厂商或框架原生能力快速覆盖；客户只在故障后临时使用。

### 路径 3：企业通用 Agent 平台

平台通常提供连接器、权限、编排、知识库和应用搭建能力。销售对象可能同时涉及业务部门、IT 和安全团队。

**潜在壁垒：** IAM/数据权限整合、私有化能力、跨部门模板和交付方法。  
**反证信号：** 收入高度依赖定制人力；每个客户都需重写大量连接逻辑。

### 路径 4：垂直 Agent

垂直产品直接围绕客服、研发、法务、财务或销售流程交付结果。

**潜在壁垒：** 专有工作流、领域评测、合规能力和结果数据闭环。  
**反证信号：** 产品只在生成文本环节提效，无法进入真实系统执行；客户可用通用 Copilot 快速替代。

## 六、项目尽调问题清单

### 技术与产品

1. 产品所称“任务完成”由谁确认？是否有业务黄金标准？
2. 与单轮 LLM、规则流程和人工基线相比，增量来自哪里？
3. 最近三个月最常见的五类失败是什么？分别由模型、工具、数据还是流程造成？
4. 更换基础模型后需要重做多少提示词、评测和客户配置？
5. 是否支持轨迹回放、状态恢复、人工审批与不可逆动作保护？
6. MCP/A2A 兼容解决了哪些成本，哪些关键连接仍需定制？

### 数据与评测

7. 评测集如何生成、更新和防止泄漏？
8. 是否保留失败轨迹和人工修改结果？这些数据能否合法用于改进产品？
9. 线上指标如何与离线评测关联？版本退化时能否自动阻断发布？
10. 客户数据如何隔离、删除和审计？

### 商业与单位经济性

11. 从签约到首个生产工作流需要多长时间、多少实施人天？
12. 收入中软件订阅、模型转售与项目实施各占多少？
13. 单个成功任务的全成本是多少？模型降价后毛利改善会由谁获得？
14. 客户扩张来自席位、用量、工作流数量还是新增部门？
15. 若客户改用另一家模型或框架，产品仍保留哪些不可替代价值？

## 七、建议建立的跟踪信号

- **技术信号：** 协议版本变化、主流框架的运行时与评测能力、长任务 benchmark、工具调用安全事件。
- **产品信号：** 从 Copilot 向可执行工作流迁移的比例、人工干预率、生产工作流数。
- **商业信号：** 上线周期、实施收入占比、净收入留存、模型成本占收入比例。
- **竞争信号：** 云厂商原生功能覆盖、开源替代速度、连接器与评测层的标准化程度。
- **风险信号：** 数据授权争议、越权调用、提示注入、错误动作的责任划分与监管要求。

## 八、结语

Agent 的核心矛盾已经从“模型能否调用工具”转向“系统能否在真实权限、数据和成本约束下稳定完成任务”。框架、协议与模型能力仍然重要，但更持久的价值可能来自运行时可靠性、评测数据、企业集成和垂直工作流。投资研究不应以 Demo 数量判断产品成熟度，而应沿着任务成功、成本、干预、风险和客户扩张五条证据链验证。
