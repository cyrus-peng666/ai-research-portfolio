# DeepGELOB：从盘口可见深度到队列状态披露

## 研究问题

标准限价订单簿通常只向模型输入价格与挂单量。相同的聚合深度背后，可能是少量大单，也可能是大量小单；挂单可能刚进入盘口，也可能已经等待较长时间。DeepGELOB 研究的问题是：订单数量、挂单年龄与事件间隔能否补充价量快照没有直接表达的队列状态。

公开代码是面向作品集的独立实现，不包含券商、交易所或课题组的原始数据、字段适配器、模型权重和非公开实验记录。

## 输入与模型

每个事件使用形状为 `(2, L, 4)` 的语义张量：

- 两侧：bid、ask；
- `L` 个盘口档位；
- 四个通道：price、volume、order count、queue age。

模型将信息拆成三条支路：

1. **PV 支路** 独立编码价格与挂单量；
2. **NA 支路** 独立编码订单数量与挂单年龄；
3. **DT 支路** 编码相邻事件的时间间隔。

三条支路在事件级融合，再由 GRU 汇总完整历史窗口。这个实现用于检验“异质微观结构变量是否应按语义组织”，不声称当前算子分配是全局最优架构。

## 严格未来收益标签

旧式平滑标签可能把输入窗口中的历史均价带入收益基准，使模型获得可从输入直接恢复的已知成分。公开实现改用严格未来均价收益：

\[
r_t = \frac{\operatorname{mean}\{m_u:t<u\leq t+h\}}{m_t}-1.
\]

目标只使用决策事件之后、同一交易日内的价格。代码在以下情况返回缺失值：

- 当日剩余数据不能覆盖完整预测期限；
- 未来期限内没有事件；
- 当前或未来价格无效；
- 目标将跨越交易日边界。

## 防止数据泄漏的约束

- `chronological_day_split` 按完整交易日划分 train、validation、test，不把同一天拆到两个集合；
- `day_aware_window_end_indices` 阻止历史窗口跨日；
- `TrainFittedStandardizer` 必须显式接收训练集 mask；
- `past_only_standardize_by_day` 只使用此前若干日的统计量，首日因缺少历史而返回缺失值；
- 激活阈值由 validation 预测校准，不能在 test 上选择。

## 评价口径

密集事件预测相邻样本高度重叠，因此代码同时报告：

- dense Pearson IC、Spearman IC、MAE、RMSE 与方向准确率；
- 按交易日重置的 time thinning / cooldown 指标；
- validation 分位数激活后的 tail signal；
- 多组 round-trip cost 假设下的净收益敏感性。

成本敏感性函数只是信号诊断，不是完整回测。它没有模拟盘口冲击、排队、成交概率、滑点和容量，因此不能据此宣称策略可交易。

## 运行公开演示

```bash
pip install -e '.[test]'
pytest tests/test_deepgelob.py
python examples/run_lob_smoke_test.py
```

演示只使用随机生成的合成数据，完成目标构造、按日切分、训练集拟合标准化、窗口生成、一次反向传播、稀疏评价和成本敏感性检查。运行结果仅用于验证代码路径，不是实证结果。

## 当前边界

1. 公开仓库没有提供受许可约束的逐笔行情，无法复刻原始样本上的数值结果。
2. 合成数据不能证明模型对真实市场有效。
3. 当前严格标签验证仍需扩展到更多股票、月份和市场状态。
4. 从预测增量到真实交易，还需使用可执行价格、延迟、手续费、滑点、冲击和容量模型。
